Inventory Drift: Why Your Numbers Stop Matching Reality

March 22, 2026 — True Good Craft Inc.

The moment you stop trusting your numbers

There is a specific point in every small shop where you stop checking the spreadsheet before pulling material. You just walk to the shelf. You stopped trusting the number because it has been wrong too many times.

The system says you have twelve sheets of acrylic. The shelf has four. Or the system says you are out of a particular fastener, but there is half a box sitting right there. The mismatch is not huge. It is just frequent enough that checking the sheet feels like a waste of time.

That gap between what the system says and what is physically there is inventory drift. It is not a catastrophic failure. It is a slow rot that changes how the shop operates without anyone formally deciding to change anything.

How drift actually happens

Drift does not come from one big mistake. It accumulates from small ones that are completely understandable in the moment.

Receiving is the most common entry point. A shipment comes in. Somebody puts it on the shelf and plans to update the sheet later. Later does not happen, or it happens with the wrong quantity, or it goes on the wrong line. The count is off from the moment the material enters the building.

Consumption is the next source. A run uses slightly more material than expected. Somebody grabs extra stock for a rework. A partial roll gets used and the remainder does not get recorded. None of this is negligence. It is just the reality of production work where recording every pull is not part of the flow.

Scrap is the quiet one. Offcuts, failed prints, damaged substrate, test pieces. Material leaves inventory without any record because there is no natural moment to log it. It was never consumed by a run. It was never sold. It just stopped existing, and the number in the system still thinks it is on the shelf.

The death spiral

Drift has a compounding problem. Once the numbers are visibly wrong, people stop updating them. Why bother entering a receipt if the starting count is already off? Why correct a balance if it will drift again next week?

So updates slow down. The gap between system and reality gets wider. Trust drops further. Eventually the system is not a record of what you have. It is a rough guess from some point in the past, and everyone knows it.

At that point the spreadsheet is not serving its original purpose. It is just a file that exists. Purchasing decisions happen from memory. Production planning happens by walking the shop. The system became optional, and nobody made that decision on purpose.

What shops do instead

When the system stops being trustworthy, shops develop workarounds. These are not written down anywhere. They just become how things work.

Physical spot checks become routine. Before starting a run, somebody walks to the shelf and counts. This works, but it means every run starts with a manual audit that should not be necessary.

Over-ordering becomes the safety net. If you cannot trust your count, you order more than you think you need. That ties up cash in excess stock and creates its own storage and tracking problems, but at least you do not run out mid-run.

The phrase "just go look" becomes a standard operating procedure. Need to know if you can take an order? Go look. Need to know if you can start a run? Go look. Need to know if it is time to reorder? Go look. The system has been quietly replaced by walking around.

None of these are wrong. They are rational responses to unreliable data. But they do not scale, and they eat time every single day.

What fixes it (before ERP)

Drift is not a problem that requires enterprise software. It requires a system where recording what happened is part of the work, not a separate chore that follows the work.

Point-of-use tracking is the core of it. When a manufacturing run consumes material, the consumption gets recorded as part of completing the run. Not after. Not in a separate step. As part of the same action. That eliminates the gap where drift enters.

Simple traceability helps too. If every run records what it consumed and what it produced, you can look back and see where a count stopped making sense. You do not need full lot tracking or serial numbers. You need enough history to see that run 47 consumed 6 metres but only 4 were deducted, and now you know where the gap came from.

Ownership of data matters more than people expect. When your inventory data lives in a local system that you control, on your machine, in a database you can query, there is no ambiguity about what the system thinks. There is no sync lag, no cloud latency, no question about which version is current. The number is right there, and if it is wrong, you can trace why.

This is exactly the kind of problem BUS Core is built to solve. Materials go in, runs consume them, stock gets adjusted from actual usage, and the ledger keeps the history. No subscription. No cloud dependency. Just a local system that keeps your numbers tied to what actually happened on the shop floor.
